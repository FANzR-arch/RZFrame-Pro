## 🏗️ 1. 结构与健康度 (Structural Integrity)
> **类比：** 建筑的主体结构。目前发现“承重墙”与“装饰层”混杂，管线（逻辑）纠缠不清。

### 🔴 [严重] "巨石"控制器 (Monolithic Controller)
- **问题描述：** `renderer.js` 虽然通过 `import` 引入为了模块，但其实仍然像是一个 **GH 巨型电池组**。第 8-36 行将所有逻辑暴力挂载到 `window` 对象上，这就像是为了方便，把家里所有电器的插头都插在一个接线板上，一旦过载（命名冲突），全屋跳闸。
- **代码位置：** `renderer.js` (Lines 8-36: `Object.assign(window, { ... })`)
- **重构逻辑：**
    - **现状**：全局变量污染。
    - **建议**：使用 React/Vue 等框架的组件状态，或至少使用 `Namespace` 封装。
    - **示例**：
      ```javascript
      // 不要直接挂 window.toggleTheme
      window.RZApp = {
        ui: { toggleTheme, resetAll },
        core: { setTemplate, exportImage }
      };
      ```

### 🟡 [建议] 样式与结构未分离
- **问题描述：** `index.html` 从第 34 行到 367 行全是 CSS 样式。这就像是把装修图纸画在了混凝土墙上，以后要换墙纸（换肤）时，还得先铲墙。
- **代码位置：** `index.html` (Line 34 `<style>`)
- **重构逻辑：** 将所有 `<style>` 内容移入 `src/styles/components.css`。

### 🟡 [建议] 变量命名“魔法数”
- **问题描述：** `updateImageScale` 中 `parseFloat(val) / 100`。虽然不算严重，但 `100` 这样的数字如果不定义常量，就像建筑图纸里不标单位也能看懂，但容易出错。
- **代码位置：** `renderer.js`

## 🌪️ 2. 健壮性与边缘风险 (Robustness & Bugs)
> **类比：** 应对台风（异常数据）的能力。目前的防御措施主要是“靠祈祷”。

### 🔴 [高危] 脆弱的异步加载 (Fragile Async)
- **问题描述：** 在 `handleFileSelect` 和 `processFiles` 中，使用了 `setTimeout` (Line 195, 248) 来强行制造延时以更新 UI。这就像是**用胶带粘住承重柱**，一旦图片处理速度快于或慢于这个 20ms/500ms，Loading 动画就会卡死或闪烁，甚至导致程序“假死”。
- **代码位置：** `renderer.js` (Line 195 `setTimeout`, Line 248 `setTimeout`)
- **重构逻辑：** 使用 `requestAnimationFrame` 配合真正的异步队列 (`async/await`)，或者使用 Web Worker 处理图片，主线程只负责渲染 UI。

### 🔴 [严重] 品牌硬编码 (Hardcoded Brands)
- **问题描述：** `tryLoadBrandLogo` (Line 70) 里面死板地写了 `'sony': 'sony'` 等映射。如果用户用了一台 "Hasselblad X2D" 但 EXIF 写的是 "Hasselblad/DJI"，由于代码里只匹配了小写，可能会匹配失败。这就像给大楼装了门禁，但必须要完全对准脸部像素才能开门。
- **代码位置：** `renderer.js` (Lines 70-85)
- **重构逻辑：** 提取为 `config/brands.json` 配置文件，并支持模糊匹配算法。

## 🛡️ 3. 安全性专项 (Electron Security)
> **类比：** 大楼的防火分区与安保系统。

### 🔴 [极危] WebSecurity Disabled
- **问题描述：** `webPreferences: { webSecurity: false }`。这是**给大楼所有的防火门都留了把钥匙**。虽然是为了加载本地图片方便，但这允许了跨域请求。如果你的应用未来加载了任何外部插件或网页，黑客可以轻松读取用户硬盘文件。
- **代码位置：** `main.js` (Line 26)
- **重构逻辑：**
    - 开启 `webSecurity: true`。
    - 使用 `protocol.registerFileProtocol` (如 `rz://`) 来安全地加载本地资源。

### 🔴 [高危] 任意文件覆盖风险
- **问题描述：** `ipcMain.handle('save-file-direct')` 接受一个 `filePath` 并直接写入。如果渲染进程被劫持（比如通过一张恶意图片注入了代码），攻击者可以让主进程把 `C:\Windows\System32\drivers\etc\hosts` 覆盖掉。
- **代码位置：** `main.js` (Line 212)
- **重构逻辑：** 严格限制写入路径，只允许写入用户显式选择的目录或 `Downloads`/`Desktop` 目录。

## ⚡ 5. 性能损耗 (Efficiency)
> **类比：** 水电能源消耗。

### 🟡 [优化] Canvas 内存泄漏风险
- **问题描述：** `updateBackground` (Line 646) 每次都 `document.createElement('canvas')` 但从未显式销毁（依赖 GC）。在批量处理 1000 张图时，这可能会瞬间吃掉大量内存，就像每喝一口水就扔一个杯子。
- **代码位置：** `renderer.js` (Line 646)
- **重构逻辑：** 使用一个复用的 OffscreenCanvas 或全局隐藏 Canvas 实例。

### 🟡 [优化] 内存中的全量图片数据
- **问题描述：** `state.images.push(imgData)` 保存了完整的图片对象。如果用户拖入 50 张 80MB 的 RAW 转 JPG，内存占用可能达到 4GB+，导致白屏崩溃。
- **代码位置：** `renderer.js`
- **重构逻辑：** 仅在内存保存缩略图和路径。原图数据随用随取，用完即焚（释放）。
