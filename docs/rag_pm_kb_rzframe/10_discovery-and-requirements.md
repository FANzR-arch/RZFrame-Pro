# 用户是谁？核心场景是什么？
RZFrame 当前服务的是“有明确交付需求的摄影用户”，而不是泛娱乐修图人群。第一类是职业摄影师，典型场景是修图完成后要快速输出一批可交付样片；第二类是摄影博主或 KOL，典型场景是在社交媒体发图时既要保留构图，也要表达拍摄参数与器材身份；第三类是胶片与器材爱好者，核心诉求是长期归档时保留参数语义和风格一致性；第四类是器材评测者，重点在于标准化展示不同机身、镜头组合。共同点是：他们已投入高成本完成“拍摄与修图”，最怕最后一步变慢、变乱、变不专业。

## 痛点清单（带证据）
1. **分享链路碎片化**：拍摄与修图后还要手工拼参数，流程断裂。证据：`docs/SHOWCASE_RZFRAME.md` 提到 Photoshop 手工加框单张约 5-10 分钟。  
2. **画质与便捷冲突**：移动端工具便捷，但专业用户担心高分图压缩、色彩损失。证据：`docs/Product_Introduction.md` 与项目自检文档强调 50MP+ 场景与无损诉求。  
3. **平台比例与构图冲突**：社媒比例规则导致原图构图被裁切。证据：`docs/项目自检/摄影边框的专业价值与市场逻辑.md` 明确“边框作为安全区”价值。  
4. **首用门槛风险**：早期文档偏开发者视角，非技术用户可能卡在入口。证据：`docs/AUDIT_DEEP_20260202.md` 对“神奇前30秒”提出风险。  
5. **专业感与身份表达不足**：用户不仅要“发图”，还要可见的专业表达与一致品牌风格。证据：`docs/SHOWCASE_RZFRAME.md` 与增长文档多次强调“品牌化封装”。

## 需求列表（用“用户故事/Job to be done”写）
1. 作为摄影师，我希望把多张照片一次性导入并批量导出，以便减少重复劳动。  
2. 作为摄影师，我希望系统自动读取 Exif 参数并展示关键字段，以便不用手工抄写。  
3. 作为器材用户，我希望系统自动识别品牌并匹配 Logo，以便快速形成专业展示。  
4. 作为创作者，我希望可以在模板内调节字体、边框、颜色、比例，以便统一视觉风格。  
5. 作为运营型创作者，我希望把当前设置应用到全部图片，以便提高出图一致性。  
6. 作为长期用户，我希望保存并导入导出模板，以便跨项目复用风格资产。  
7. 作为隐私敏感用户，我希望所有处理在本地完成，以便避免上传原图风险。  
8. 作为多语言用户，我希望可切换 CN/EN/JP，以便降低学习成本。  
9. 作为进阶用户，我希望异常情况下有兜底方式（如 Logo 缺失时文本回退），以便不中断交付。  
10. 作为产品经理，我希望拿到可追踪的指标闭环，以便判断功能优先级是否正确【待补充】。

## 优先级与取舍（RICE/影响力-成本/风险矩阵均可，用文字解释即可）
当前优先级策略可归纳为“影响力优先 + 成本可控 + 风险可兜底”。  
P0 聚焦导入、解析、模板预览、导出、批处理、自动 Logo 等核心闭环，因为这些能力直接决定用户是否能完成任务。  
P1 放在模板资产化、多语言、主题切换、Logo 持久化等增强能力，目标是提高复用率与留存，但不阻塞首用价值。  
P2 或未来项包括云同步、社区、社媒直发、AI 配色、地理可视化等，这些方向潜在价值高，但成本和不确定性显著更高，且会拉长当前迭代周期。  
取舍逻辑是：先确保“能稳定交付”，再做“更好看、更智能、更联网”。

## 未做的需求（为什么不做）
1. **账号体系与云同步**：当前定位是本地优先工具，先保证隐私和稳定性；云化会带来开发、运维与合规成本。  
2. **一键社媒发布**：跨平台接口和审核策略复杂，短期 ROI 不清晰，且偏离“本地封装”主目标。  
3. **AI 自动配色与智能推荐**：在路线图中，但目前缺少可验证的用户收益数据与质量基线。  
4. **更广格式与深度色彩管理能力**：如 TIFF/HEIC/ICC 深度处理仍在规划，属于专业能力升级而非首发必需。  
5. **完整商业化闭环（付费墙与支付）**：已有策略草案，但真实定价、转化与渠道数据仍需验证【待补充】。
